Operating System Fundamentals

OS CORE
What is an Operating System? What are its 5 core responsibilities?
Google Amazon Razorpay
★★★ ▼

An OS is system software that manages hardware resources and provides a uniform interface for applications. Think of it as a resource manager + abstraction layer between raw hardware and user programs.

Process MgmtCreate, schedule, pause, kill processes. Allocate CPU time.
Memory MgmtVirtual memory, paging, swapping. Isolate process memory.
File SystemOrganize data on storage. Inodes, directories, permissions.
I/O MgmtAbstract hardware via device drivers. Buffering, caching.
SecurityAuthentication, permissions, system call gating.
💡 Answer Pattern for FAANG
Don't just list — explain the why: "The OS abstracts hardware complexity so application developers don't need to know if the disk is SSD, HDD, or NVMe. It enforces isolation so a buggy process can't corrupt another process's memory."
Explain Kernel Space vs User Space. What happens during a system call?
Google Amazon Goldman
★★★ ▼

User Space (Ring 3): Where your application runs. Can't directly touch hardware. Must ask the kernel via syscalls. One crash = just that process dies.

Kernel Space (Ring 0): Privileged mode. Full hardware access. All OS code runs here. One bug can crash the whole system.

System Call internal flow:

# e.g. read(fd, buf, 100) in your C program
1. glibc wrapper puts syscall number in RAX register (e.g. RAX=0 for read)
2. Executes SYSCALL instruction
3. CPU switches Ring 3 → Ring 0 (hardware protection level)
4. CPU saves user registers, jumps to kernel's syscall handler
5. Kernel validates args, does the real work
6. Result placed in RAX, CPU executes SYSRET
7. Ring 0 → Ring 3, your app continues
# Cost: ~100-500 ns per syscall
⚡ HFT / Fintech Angle — Critical to Know
Each syscall costs ~100–500ns minimum. That's why HFT systems use kernel bypass (DPDK), memory-mapped I/O, and busy-polling instead of blocking syscalls. Goldman/Citadel ask this to probe whether you understand why their systems avoid standard Linux networking.
Monolithic vs Microkernel vs Hybrid kernel — with real examples. Which does Linux use?
AmazonMicrosoft
★★ ▼
Type Services in kernel? Speed Stability Example
Monolithic Everything (FS, drivers, net) Fastest A bug can crash OS Linux, Unix
Microkernel Only IPC, scheduling, memory Slower (IPC overhead) Very stable Mach, MINIX, QNX
Hybrid Core in kernel, some services outside Good balance Good Windows NT, macOS XNU

Linux = Monolithic BUT Modular: You can load/unload kernel modules (insmod, rmmod, lsmod) at runtime without rebooting. Best of both worlds.

Follow-up Amazon asks
"Why is Linux monolithic if microkernel is theoretically more stable?" → Answer: Performance. The IPC overhead of a microkernel makes it impractical for general-purpose OSes. Linux compensates with modules and good driver isolation.
What is a context switch? What is its true cost and what triggers it?
Google Citadel Goldman
★★★ ▼

A context switch = CPU stops running process A, saves its state, loads process B's state, runs B. The kernel maintains a PCB (Process Control Block) per process.

What gets saved/restored in PCB: all CPU registers (GPRs, PC, SP, flags), page table pointer (CR3), open file descriptor table, signal masks, scheduling info.

What triggers it:

PreemptionScheduler time quantum expires (CFS)
I/O BlockProcess calls read() and blocks waiting for disk/network
Higher PriorityReal-time task becomes runnable (SCHED_FIFO)
Voluntarysched_yield(), sleep(), mutex lock

True cost breakdown — why it's expensive:

# Direct cost:
Save/restore registers:     ~200-500 ns
TLB flush (if no PCID):    ~500 ns - 5 µs (worst case)
# Indirect cost (the real killer):
CPU cache (L1/L2/L3):      Now cold for new process → cache misses
Branch predictor:          Flushed → mispredictions spike
Total observable cost:     1 µs - 100 µs depending on cache pollution
# Measure context switches
vmstat 1          # 'cs' column
pidstat -w 1      # per-process voluntary/involuntary
perf stat ./app   # 'context-switches' counter
★ Citadel / Goldman Key Point
High context switch rate is a latency killer. HFT uses SCHED_FIFO + CPU pinning (taskset / isolcpus) to eliminate involuntary switches entirely for trading threads.

Interview Question: What is SUID?
SUID (Set User ID) bit means the program runs with the file owner's privileges, not the caller's. Example: passwd command is owned by root with SUID set — so any user can run it to change their own password, but internally it modifies /etc/shadow which only root can write. Security risk if set on wrong binary.

Processes & Threads — Most Asked at FAANG

OS CORE
Process vs Thread — ALL differences. What's shared, what's not?
Google Amazon Goldman
★★★ ▼
Aspect Process Thread
Address Space Own separate address space Shares parent process's address space
Memory (Heap) Own heap Shared heap (race conditions possible)
Stack Own stack Each thread has own stack
File Descriptors Own copies after fork() Shared — close in one = affects all
Creation Cost Heavy — fork() copies page tables Light — clone() with shared mm
Context Switch Expensive — TLB flush, new page tables Cheaper — same address space
Crash Impact Isolated — other processes survive One thread crashes = entire process crashes
Communication IPC needed (pipes, sockets, shm) Direct shared memory (fast but unsafe)
Linux impl fork() → creates task_struct clone(CLONE_VM|CLONE_FS|...) → task_struct
Key Insight — Linux "Threads" are just Processes
In Linux, there is no separate "thread" type. Both processes and threads use the same task_struct. The difference is only in which resources they share — controlled by flags passed to clone(). POSIX's pthread_create() ultimately calls clone() with sharing flags.
What is fork()? Explain Copy-on-Write. What are Zombie and Orphan processes?
AmazonGoogleMicrosoft
★★★ ▼

fork(): Creates exact copy of calling process. Returns 0 to child, child's PID to parent. Uses Copy-on-Write (CoW).

Copy-on-Write (CoW): After fork(), parent and child share the same physical memory pages — marked read-only. When either tries to write, a page fault fires, kernel copies just that page to a private copy. Only pages actually written get duplicated. Makes fork() fast despite "copying" a huge process.

# Classic shell pattern: fork + exec
pid = fork()
if pid == 0:          # child process
    exec("/bin/ls", args)   # replace child with new program
else:                  # parent
    wait(&status)           # wait for child, collect exit status
Zombie 🧟Child exited, parent hasn't called wait(). PCB stays in /proc. Fix: call wait() or handle SIGCHLD signal.
Orphan 👶Parent died first, child still running. Linux re-parents to init/systemd (PID 1), which periodically calls wait().
# Find zombies
ps aux | grep 'Z'              # STAT column = Z
ps -eo pid,ppid,stat,cmd | grep Z
Explain all Linux signals. What is SIGKILL vs SIGTERM? How do you gracefully shutdown a service?
GoogleAmazonGoldman
★★★ ▼
Signal Num Catchable? Meaning & Use
SIGTERM 15 ✅ Yes Graceful termination request. App can cleanup, flush buffers, close DB connections before exiting. ALWAYS try this first.
SIGKILL 9 ❌ No Forceful instant kill by kernel. Cannot be caught or ignored. Data may be lost. Use only if SIGTERM doesn't work.
SIGHUP 1 ✅ Yes Hangup. Daemons (nginx, sshd) use it to reload config without restart: kill -HUP <pid>
SIGINT 2 ✅ Yes Interrupt — Ctrl+C in terminal
SIGSEGV 11 ✅ Yes Segmentation fault — invalid memory access. Default: dump core + terminate.
SIGCHLD 17 ✅ Yes Child process changed state. Parent should call wait() to prevent zombies.
SIGSTOP 19 ❌ No Pause process. Cannot catch. Ctrl+Z sends SIGTSTP (which IS catchable).
SIGALRM 14 ✅ Yes Timer expired (from alarm() syscall). Used for timeouts. 
# Graceful shutdown pattern (Kubernetes, microservices)
kill -15 <pid>         # 1. Send SIGTERM — let it cleanup
sleep 30               # 2. Wait up to 30 seconds
kill -9 <pid> 2>/dev/null  # 3. Force kill if still running

# In shell script — trap signals
trap 'echo "Cleaning up..."; rm -f /tmp/lockfile; exit 0' SIGTERM SIGINT
Race conditions, mutex, semaphore, spinlock — explain each and when to use which.
GoogleCitadelJane St
★★★ ▼

Race Condition: Two threads read-modify-write shared data. Outcome depends on execution order. Example: counter++ is actually 3 operations (read, add, write) — not atomic.

MutexBinary lock. Only one thread can hold it. Blocks others. OS schedules away waiting thread (no CPU waste). Use for: most shared data protection.
SemaphoreCounting lock. N threads can hold it simultaneously. Great for: "max 10 DB connections at once".
SpinlockThread loops (busy-waits) checking lock. Uses 100% CPU while waiting. Use only for: very short critical sections where sleep overhead > wait time. HFT loves these.
RW LockMultiple readers OR one writer. Perfect for: read-heavy (order book reads), rare writes (config updates).
Cond. VarThread waits for a condition. Always used with mutex. pthread_cond_wait releases mutex while waiting — avoids deadlock.
# Mutex example (C)
pthread_mutex_lock(&m);
shared_counter++;         # critical section
pthread_mutex_unlock(&m);

# Atomic ops (lock-free, fastest)
__atomic_fetch_add(&counter, 1, __ATOMIC_SEQ_CST);
std::atomic<int> cnt; cnt.fetch_add(1);
Deadlock = 4 Conditions (Coffman, 1971)
Mutual Exclusion + Hold & Wait + No Preemption + Circular Wait. ALL four must hold. Break any one to prevent deadlock. Most common fix: consistent lock ordering (always acquire lock A before lock B, everywhere in code).

Memory Management

OS CORE
Explain Virtual Memory and Paging. How does address translation work? What is TLB?
GoogleAmazonGoldman
★★★ ▼

Virtual Memory gives every process the illusion of a private, large address space (0 to 2^64 on 64-bit). Actual physical RAM is much smaller. The OS + hardware MMU translate virtual → physical transparently.

# Virtual Address (x86_64, 4-level paging)
VA: [PML4 (9 bits)] [PDPT (9)] [PD (9)] [PT (9)] [Offset (12)]
         ↓               ↓          ↓        ↓
     Page Dir 4    → Page Dir 3 → PD → Page Table → Physical Frame + Offset

# TLB = cache of recent VA → PA translations
# TLB hit:  1-2 CPU cycles
# TLB miss: full 4-level walk = 4 memory accesses = ~100+ cycles
Page FaultVA not mapped to physical frame. Kernel's fault handler runs: allocate frame, load page, update page table, retry instruction.
Major FaultPage must be loaded from disk (swap). Slow — involves I/O ~1-100ms. Kills latency.
Minor FaultPage in memory but not mapped yet (e.g. shared lib first access). No disk I/O. Fast.
Huge Pages2MB or 1GB pages. Fewer TLB entries needed. Massive perf win for large in-memory DBs. Used by Redis, Oracle, HFT.
# Check TLB-related perf
perf stat -e dTLB-misses,iTLB-misses ./app
cat /proc/meminfo | grep Huge
echo 1024 > /proc/sys/vm/nr_hugepages   # allocate huge pages

AmazonGoogleRazorpay

Aspect Stack Heap

What is mmap()? How does it work and why do fintech systems use it extensively?
GoldmanCitadelRazorpay
★★ ▼

mmap() maps a file (or anonymous memory) directly into the process's virtual address space. Once mapped, accessing ptr[i] is reading/writing the file — no explicit read()/write() syscalls needed. Kernel handles I/O lazily via page faults.

# Map a file into memory
int fd = open("market_data.bin", O_RDWR);
void* ptr = mmap(NULL, file_size, PROT_READ|PROT_WRITE, MAP_SHARED, fd, 0);
# Now ptr[0..size-1] IS the file. Write to ptr → writes to file
msync(ptr, size, MS_SYNC);   # flush changes to disk
munmap(ptr, size);           # unmap
Zero-copy I/ONo kernel↔user buffer copies. Huge win for large sequential reads (market data replay).
Shared Memory IPCTwo processes mmap same file → shared memory. LMAX Disruptor uses this for inter-process ring buffers.
LMDB / RocksDBDatabases that mmap entire data files for zero-copy reads. Much faster than read() syscalls.
mlock()Pin mmap'd pages in RAM, prevent swapping. Critical for latency-sensitive fintech. mlockall() for entire process.

CPU Scheduling

OS CORE
How does Linux CFS (Completely Fair Scheduler) work? Red-Black Tree, vruntime.
GoogleGoldmanCitadel
★★★ ▼

CFS (since Linux 2.6.23) replaces the old O(1) scheduler. Core idea: give every process a fair share of CPU proportional to its priority weight.

vruntime (virtual runtime): Each task has a vruntime counter that ticks proportionally to CPU time used, weighted inversely by priority. Lower nice value = slower vruntime growth = more CPU time.

Red-Black Tree: All runnable tasks stored in an RB-tree sorted by vruntime. O(log n) insert/delete. Leftmost node = task with smallest vruntime = runs next. O(1) to get next task.

# Priority (nice) values
nice -n -20 ./critical-app   # highest priority (gets ~40% more CPU)
nice -n 19 ./batch-job        # lowest priority
renice -5 -p <pid>           # adjust live process priority
Policy Type Use Case
SCHED_OTHER Default CFS Regular applications
SCHED_FIFO Real-time (no preemption by same-priority) HFT trading engine, audio, control systems
SCHED_RR Real-time with time quantum Real-time with fairness among RT tasks
SCHED_BATCH CPU-intensive, no boost Compiles, batch processing
SCHED_DEADLINE EDF with hard deadlines Real-time systems requiring guaranteed latency 
# Set real-time scheduling (needs root)
chrt -f 99 ./hft-trading-engine    # SCHED_FIFO priority 99 (max)
chrt -p <pid>                      # check current policy

Deadlock, Synchronization & Concurrency

OS CORE
Deadlock — 4 conditions, prevention strategies, detection in Linux.
AmazonGoogleGoldman
★★★ ▼

Deadlock: Thread A holds lock 1, waits for lock 2. Thread B holds lock 2, waits for lock 1. Both wait forever.

Coffman's 4 Necessary Conditions — ALL must hold:

1. Mutual Excl.Resource can only be held by one thread at a time
2. Hold & WaitThread holds a resource while waiting for another
3. No PreemptionResources can't be forcibly taken away
4. Circular WaitCircular chain: A→B→C→A each waiting for next

Prevention (break any one condition):

Lock OrderingAlways acquire locks in same global order everywhere in code. Breaks circular wait. Most practical solution.
Try-Lock + Timeoutpthread_mutex_trylock() — don't block, return EBUSY. Or timed lock with timeout.
Lock-FreeUse atomic CAS operations. No locks = no deadlock. Used in HFT.
Request All at OnceAcquire all needed locks at start (breaks Hold & Wait).
# Detect deadlock in Linux
pstack <pid>                    # print thread stacks of live process
gdb -p <pid> -ex "thread apply all bt"  # all thread backtraces
cat /proc/<pid>/wchan           # kernel function thread is sleeping in
valgrind --tool=helgrind ./app  # detect races + potential deadlocks
What is a lock-free data structure? Explain CAS (Compare-And-Swap) and ABA problem.
CitadelJane StGoldman
★★★ ▼

Lock-free: Thread-safe without mutexes. Uses hardware atomic instructions (CAS). No blocking = no deadlock, no priority inversion, better scalability.

CAS — Compare-And-Swap: Atomic instruction: "If *ptr == expected, set *ptr = new_val and return true. Otherwise return false." The foundation of all lock-free algorithms.

// C++ atomic CAS
std::atomic<int> counter(0);
int expected = 5;
bool ok = counter.compare_exchange_strong(expected, 10);
// If counter was 5, now it's 10. If not, expected holds actual value.

// Lock-free increment pattern:
int old, next;
do {
    old = counter.load();
    next = old + 1;
} while (!counter.compare_exchange_weak(old, next));

ABA Problem: Value changes A→B→A. CAS sees A both times and thinks nothing changed — but the state between the reads DID change. Example: Node removed and re-inserted in lock-free linked list.

Fix: Add a version counter alongside the value: instead of comparing just the value, compare value+version. Version always increments. Java: AtomicStampedReference. C++: Tagged pointer.

Memory Ordering (Interview Deep-Dive)
memory_order_relaxed: No ordering guarantee, fastest. memory_order_acquire/release: One-way barriers (pub-sub pattern). memory_order_seq_cst: Full fence, safest, default. HFT uses relaxed where safe, seq_cst at critical points.

File Systems & Linux Internals

LINUX
What is an inode? What's the difference between hard links and soft links?
AmazonGoogleMicrosoft
★★★ ▼

An inode (index node) is a data structure storing ALL metadata about a file — except its name and content. Every file has exactly one inode.

Inode storesFile type, permissions (rwx), UID/GID, file size, timestamps (atime/mtime/ctime), link count, block pointers to data
NOT in inodeFilename (stored in directory entry/dentry) and file content (stored in data blocks)

Hard Link: Another directory entry pointing to the same inode. File deleted only when link count reaches 0. Can't span filesystems.

Soft/Symbolic Link: A special file containing a path string to another file. Different inode. Can span filesystems. Breaks if target deleted.

# Example
ln /etc/nginx/nginx.conf /backup/nginx.conf   # hard link (same inode!)
ln -s /opt/java17/bin/java /usr/bin/java      # symlink (like a shortcut)
ls -i file.txt        # show inode number
stat file.txt         # full inode info
df -i                 # inode usage (can run out even with disk space!)
Classic Interview Trap
"My disk has space but I can't create files!" → inode table is full. Check: df -i. Happens on servers with millions of small files (email servers, log archives).
What is blocking I/O vs non-blocking vs async? Explain epoll — why is it O(1)?
GoogleAmazonGoldman
★★★ ▼
Model Thread blocks? How Used in
Blocking I/O Yes — sleeps until data ready Default read()/write() Simple servers, databases
Non-blocking I/O No — returns EAGAIN immediately O_NONBLOCK flag With select/poll/epoll
I/O Multiplexing Yes (in epoll_wait) Monitor many FDs, wake when any ready Nginx, Redis, Node.js
Async I/O (io_uring) Never Submit + completion ring buffers Modern high-perf servers

Why epoll beats select/poll:

# select/poll: O(n) scan ALL registered FDs each call
# 100k connections = check 100k FDs even if 1 is ready

# epoll internals:
epoll_create()         # creates: RB-tree (all monitored FDs) + ready list
epoll_ctl(ADD, fd)     # adds fd to RB-tree: O(log n)
# When NIC receives data → IRQ → driver callback marks fd ready
# → moves fd from RB-tree to ready list: O(1) per event
n = epoll_wait(...)    # returns ONLY ready FDs: O(k) where k=ready count
# Total: O(1) regardless of total connections!
Edge Trigger (ET) vs Level Trigger (LT)
LT (default): Notifies as long as data is available. Easy but may notify many times. ET: Notifies only when state changes. Must drain all data in one go. Higher performance, complex. Nginx uses ET mode.
Linux namespaces and cgroups — how do they form the foundation of Docker/containers?
AmazonGoogle
★★★ ▼

Namespaces = what a process can see. Each namespace type creates an isolated view of a different resource:

PID nsContainer sees its own PID tree starting at 1. Its PID 1 = actual PID 12345 on host.
NET nsOwn network interfaces, IP, routing, firewall rules. Container eth0 ≠ host eth0.
MNT nsOwn filesystem mount table. Container sees its own / (root).
UTS nsOwn hostname. Container can be named "app1" independently.
USER nsMap container root (uid 0) to unprivileged host uid 10000. Rootless containers.

cgroups (Control Groups) = what a process can use:

cpuLimit CPU time: max 50% of one core
memoryHard RAM limit + OOM behavior

Shell scripting: write a script to monitor a service and restart if it crashes.
AmazonRazorpay
★★ ▼ 
#!/bin/bash
set -euo pipefail    # -e=exit on error, -u=undefined var=error, -o pipefail

SERVICE="payment-gateway"
LOG="/var/log/watchdog.log"
MAX_RETRIES=5; retries=0

log() { echo "$(date '+%F %T') [$1] $2" | tee -a "$LOG"; }

while true; do
    if systemctl is-active --quiet "$SERVICE"; then
        log INFO "$SERVICE is running (PID: $(systemctl show -p MainPID --value $SERVICE))"
        retries=0
    else
        log WARN "$SERVICE is DOWN. Attempting restart... (attempt $((retries+1))/$MAX_RETRIES)"
        if [[ $retries -ge $MAX_RETRIES ]]; then
            log ERROR "Max retries reached. Sending alert..."
            # curl -X POST https://alerts.company.com/webhook -d "service down"
            break
        fi
        systemctl restart "$SERVICE"
        ((retries++))
    fi
    sleep 30    # check every 30 seconds
done

# Real world: use systemd Restart=always + RestartSec=5 instead!
Interviewer Follow-up: Why not just use systemd?
Answer: In production, always prefer Restart=always in the systemd unit file — it's more reliable, integrated, and handles edge cases. Shell watchdog scripts are brittle. But knowing how to write one shows scripting competence.

SCENARIO: "Too many TIME_WAIT sockets on our payment API server — what do you do?"
GoldmanStripeRazorpay
★★★ ▼

Why TIME_WAIT happens: After TCP active closer sends final ACK, it waits 2×MSL (60–120 sec) before port becomes reusable. This prevents stale packets from a previous connection confusing a new one on the same port.

Why it's a problem for payment APIs: Each HTTP request opens a new socket. With thousands of req/sec, TIME_WAIT sockets accumulate. Port range (default 32768–60999 = 28k ports) gets exhausted. New connections fail with "Cannot assign requested address".

# Diagnose
ss -s | grep TIME-WAIT
ss -o state time-wait | wc -l   # count

# Solutions (from best to worst)
# 1. Use HTTP Keep-Alive / connection pooling (reuse connections!)
# 2. Expand ephemeral port range
sysctl -w net.ipv4.ip_local_port_range="1024 65535"

# 3. Allow TIME_WAIT socket reuse for new outbound connections
sysctl -w net.ipv4.tcp_tw_reuse=1      # safe for most cases

# 4. Reduce TIME_WAIT duration (not recommended usually)
sysctl -w net.ipv4.tcp_fin_timeout=15  # default 60s

# Make persistent
echo "net.ipv4.tcp_tw_reuse=1" >> /etc/sysctl.conf
sysctl -p
Root Fix
The real solution is always connection pooling + HTTP Keep-Alive. Tune kernel params as a band-aid only. A payment gateway using connection pools (HikariCP for DB, HTTP client connection pools for APIs) will never hit TIME_WAIT exhaustion.

Performance Tuning & Debugging

LINUX
What is the OOM Killer? How does Linux decide which process to kill?
AmazonGoogleGoldman
★★ ▼

When the system runs out of physical + swap memory, Linux's OOM (Out-of-Memory) Killer selects and kills a process to free memory. The goal is to maximize freed memory while minimizing damage.

Score calculation: Each process has an oom_score (0–1000). Higher score = more likely killed. Main factors: RSS (memory used), child process memory, runtime (longer-running processes get slight protection), nice value.

# View and adjust OOM scores
cat /proc/<pid>/oom_score         # current score (0-1000)
cat /proc/<pid>/oom_score_adj     # adjustment knob (-1000 to 1000)

# Protect critical service (trading engine, payment processor)
echo -1000 > /proc/<pid>/oom_score_adj   # NEVER kill this
echo 1000 > /proc/<pid>/oom_score_adj    # kill this first

# In systemd unit file:
OOMScoreAdjust=-500

# Detect OOM kills in logs
dmesg | grep -i "oom\|killed process"
journalctl -k --since "1 hour ago" | grep -i oom
Prevention Strategy
Set memory limits via cgroups (Docker: --memory=4g). Use vm.overcommit_memory=2 to prevent kernel from accepting more memory commitments than available. Always set OOMScoreAdjust=-500 in systemd unit for critical services like order matching engines.

How would you tune a Linux server from OS level to achieve sub-100µs trade execution latency?
CitadelGoldmanJane St
★★★ ▼

This is the ultimate OS interview question for HFT. Show systematic thinking across all layers:

# 1. CPU Isolation — prevent OS tasks from interfering
# Add to /etc/default/grub GRUB_CMDLINE_LINUX:
isolcpus=2,3,4,5 nohz_full=2,3,4,5 rcu_nocbs=2,3,4,5
# isolcpus: scheduler won't assign tasks to these cores
# nohz_full: disable scheduler tick on isolated cores
# rcu_nocbs: disable RCU callbacks on isolated cores

# 2. CPU Frequency — disable power saving
echo performance > /sys/devices/system/cpu/cpu*/cpufreq/scaling_governor
# Prevents frequency scaling latency spikes (C-state exits: 1-100µs)

# 3. NUMA-aware placement
numactl --cpunodebind=0 --membind=0 ./trading-engine
# Remote NUMA memory access is 2-3x slower

# 4. Real-time scheduling
chrt -f 99 taskset -c 2 ./order-engine    # SCHED_FIFO on isolated core
echo -1000 > /proc/$(pgrep order-engine)/oom_score_adj  # never OOM kill

# 5. Memory — prevent swapping and page faults
sysctl -w vm.swappiness=0             # disable swap
# In app: mlockall(MCL_CURRENT | MCL_FUTURE)  ← lock ALL pages in RAM
echo 1024 > /proc/sys/vm/nr_hugepages  # 2MB huge pages → fewer TLB misses

# 6. Network — reduce latency
sysctl -w net.ipv4.tcp_nodelay=1       # disable Nagle's algorithm
sysctl -w net.core.busy_poll=50        # busy-poll for 50µs before sleep
# IRQ affinity — pin NIC interrupts to non-isolated cores
echo 1 > /proc/irq/<NIC_IRQ>/smp_affinity

# 7. Disable unnecessary jitter sources
systemctl disable bluetooth avahi-daemon cups NetworkManager
# Disable CPU microcode updates during boot if testing latency
Order of impact (biggest to smallest)
1. CPU isolation + SCHED_FIFO (~70% of benefit) · 2. mlockall() + huge pages (TLB misses) · 3. NUMA binding · 4. TCP tuning · 5. Misc (disable C-states, performance governor)
Explain DPDK and kernel bypass networking. What's wrong with standard Linux networking for HFT?
CitadelGoldmanJane St
★★ ▼

Standard Linux network path latency breakdown:

NIC → IRQ → NAPI softirq → kernel TCP stack → socket buffer
→ epoll_wait wakeup → syscall overhead → copy to userspace
Total: 30–100 µs minimum (unacceptable for µs-latency HFT)

What kills latency in standard path:

IRQ handlingHardware interrupt + context switch: ~1-5µs
Kernel TCP stackChecksums, routing, socket buffers: ~10-30µs
Memory copiesNIC DMA → kernel buffer → user buffer: ~1-5µs
Syscall overheadMode switch + wakeup: ~1-2µs

DPDK (Data Plane Development Kit) solution:

# DPDK moves NIC driver to user-space
# App busy-polls NIC ring buffer directly — NO interrupts!
# Zero copies: NIC DMA directly to huge-page mapped user memory
# Zero syscalls: reads directly from memory-mapped ring buffer

Normal:   NIC → kernel → copy → user app        ~50-100 µs
DPDK:     NIC → user-space ring buffer           ~1-5 µs
RDMA:     Remote NIC → local app memory (HW)    <1 µs
io_uring — Middle Ground (Linux 5.1+)
Not as fast as DPDK but much faster than classic sockets. Uses two ring buffers in shared memory between kernel and userspace. Submission and completion queues. Zero syscalls for the common case. Used in RocksDB, NGINX, Ceph. Ideal for ultra-low latency without kernel bypass complexity.

Linux boot process end-to-end: BIOS → GRUB → Kernel → systemd → your app.
AmazonGoogleGoldman
★★ ▼ 
1. BIOS / UEFI
   └─ POST: Power-On Self Test, initialize CPU, RAM, PCI devices
   └─ UEFI: scans for EFI System Partition (FAT32, /boot/efi)
   └─ Loads bootloader: /boot/efi/EFI/ubuntu/grubx64.efi

2. GRUB2 Bootloader
   └─ Reads /boot/grub/grub.cfg
   └─ Shows boot menu (default 5s timeout)
   └─ Loads kernel image: /boot/vmlinuz-6.x.x
   └─ Loads initrd: /boot/initrd.img (compressed ramdisk)
   └─ Passes kernel cmdline: root=/dev/sda1 ro quiet isolcpus=2,3

3. Kernel Initialization
   └─ Decompresses itself (vmlinuz = zlib-compressed kernel)
   └─ Detects CPU, sets up GDT/IDT, enables paging/MMU
   └─ Initializes memory manager, scheduler, device drivers
   └─ Mounts initramfs as temporary root filesystem

4. initramfs / initrd
   └─ Contains minimal tools: busybox, storage drivers (NVMe, LVM, RAID)
   └─ Mounts real root filesystem (may need to decrypt LUKS, assemble RAID)
   └─ Executes /init script, pivots to real root

5. systemd (PID 1)
   └─ Reads /lib/systemd/system/*.target files
   └─ Resolves dependencies, starts services in parallel (vs init's serial)
   └─ Mounts /proc /sys /dev /run (virtual filesystems)
   └─ Reaches multi-user.target → networking, sshd, app services start

6. Your application starts
   └─ systemd spawns your service per unit file
   └─ Applies cgroup limits (CPU, memory from unit file)
   └─ App is ready to serve traffic
